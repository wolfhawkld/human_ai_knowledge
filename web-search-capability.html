🔍 Web Search 能力总结

模型层 vs 工具层 | 生成者: Nemo | 2026-03-10

核心洞察：存在两层搜索能力——模型层搜索（内置联网）和工具层搜索（OpenClaw 工具），两者互补但机制不同。

两层搜索能力

层级 定义 控制方 特点
模型层搜索 模型内置的联网能力 模型服务商 自动触发、不可控、OpenClaw 不可见
工具层搜索 OpenClaw 的 web_search / web_fetch OpenClaw / 用户 显式调用、可控、可配置参数

模型 模型层搜索 工具层支持 中国可用 说明
GPT-4/4o ✅ Browsing ✅ 强 ❌ 需代理 function call 能力强
Claude ✅ Web Search ✅ 强 ❌ 需代理 工具调用能力优秀
Gemini ✅ Grounding ✅ 原生 ❌ 需代理 OpenClaw 可直接配置

DeepSeek ✅ 内置联网 ✅ 支持 ✅ 直接访问 工具调用能力较好

OpenClaw 集成策略

方案一：纯模型层搜索

用户问题 → 模型 → 模型自动联网 → 返回结果

适合快速问答，不需要精确控制。

方案二：工具层搜索

用户问题 → OpenClaw 判断 → 调用 web_search → 结果注入 prompt → 模型回答

可控、可观察、可指定参数。配置方式：openclaw configure --section web

方案三：混合模式（推荐）

用户问题 → OpenClaw 尝试工具搜索 → 失败则依赖模型层搜索

中国大陆推荐配置

无代理环境

{
  tools: {
    web: {
      search: {
        provider: "kimi",
        kimi: { apiKey: "your-kimi-api-key" }
      }
    }
  }
}

有代理环境

{
  tools: {
    web: {
      search: {
        provider: "brave",
        apiKey: "BRAVE_API_KEY"
      }
    }
  }
}

web_search vs web_fetch

工具 功能 需要配置 使用场景
web_search 搜索网页，返回结果列表 需 API key "最近 AI 新闻"
web_fetch 抓取指定 URL 内容 默认启用 "帮我读这篇文章"

关键洞察

  1. 模型联网 ≠ 工具调用：很多模型有内置联网，但不一定能响应 OpenClaw 的工具调用
  2. GLM 的特殊性：内置联网强，但工具调用弱——适合"直接问"，不适合复杂工具编排
  3. Claude/GPT 优势：工具调用能力强，配合 OpenClaw 工具效果最佳
  4. 国产方案排序：Kimi > DeepSeek > Qwen > GLM（工具调用能力）
  5. 搜索质量排序：Brave/Perplexity > Kimi > 模型内置搜索

实践建议

场景 推荐方案
快速问答，不在乎精确控制 用 GLM/Kimi，依赖模型层搜索
需要控制搜索参数（时间、地区） 配置 web_search 工具
复杂任务（搜索 + 读取 + 分析） Claude/GPT + 工具链
中国大陆无代理 Kimi 作为 provider，或 GLM 模型层搜索
追求搜索质量 Brave/Perplexity（需代理）
一句话总结
模型层搜索省心但不可控，工具层搜索可控但需配置；
中国大陆无代理首选 Kimi，有代理首选 Brave/Perplexity。
