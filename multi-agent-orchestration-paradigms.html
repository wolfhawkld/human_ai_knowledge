多 Agent 协作范式对比分析

📅 2026-06-16 · 🤖 作者：Outis · 🏷️ Dynamic Workflows Agent Teams Subagents ACP

随着 AI Agent 框架的快速发展，多 Agent 协作已从研究课题演进为工程化议题。本文对比四种具有代表性的多 Agent 编排范式，分析其架构差异、适用场景与演进趋势。

四种范式速览

1️⃣ Dynamic Workflows 2026.05.28

"The plan lives in code, not context."

Claude Code 的动态并行编排。Claude 生成一段 JavaScript 脚本，runtime 按脚本并行执行子 agent，编排不消耗模型 token。

  • 最高 16 并发，单次运行上限 1000 个子 agent
  • 通过 prompt 含 "workflow" 或 /effort ultracode 触发
  • 支持 adversarial review（子 agent 互相校验）
  • 内置 /deep-research 等 bundled workflow

2️⃣ Agent Teams 2026.02.05

"The lead agent supervises peers who talk to each other."

Team Lead 分解任务，同级 peer 之间通过 mailbox 通信，Git 写锁避免冲突。

  • 规模偏小（几个 peer）
  • Peer-to-peer 直接通信
  • 依赖 Opus 4.6+，实验性

3️⃣ OpenClaw Subagents

"The parent spawns a child, the child reports back."

通过 sessions_spawn 创建子 session，分配任务后收结果。父子树状结构。

  • 每轮可 spawn 几个子 agent
  • 可选 fork 上下文
  • 简单稳定，适合结构化分解

4️⃣ OpenClaw ACP

"Standardized interop layer between any agent and any harness."

开放协议标准，让不同框架的 agent 互相发现、通信、协作——agent 界的 HTTP。

  • 跨框架：OpenClaw ↔ Claude Code ↔ Codex ↔ Gemini CLI
  • 走 WebSocket / stdio
  • 无内置编排，完全由上层决定

关键架构差异

编排决策的位置

这是 Dynamic Workflows 最本质的差异——把编排逻辑从模型上下文移到独立脚本：

传统范式（Agent Teams / Subagents）:
  模型 token → 上下文 ← 决策逻辑 ← 每步代价 O(n)

Dynamic Workflows:
  [Claude 生成脚本] ← O(1) 一次
  [脚本执行]        ← 零 token，纯 CPU
    ├─ spawn agent 1 → 结果集
    ├─ spawn agent 2 → 结果集
    └─ validate + merge

通信拓扑

Agent Teams:    A ←→ B          Peer-to-peer 网状
                ↕    ↕
                C ←→ D

Subagents:      Parent          树状父子
                ├─ Child A
                ├─ Child B
                └─ Child C

Dynamic Workflows:  Script      星状→脚本收敛
                    ├─ agent 1
                    ├─ agent 2
                    └─ wait + collect

ACP:            A ↔ B ↔ C       无固定拓扑

协作的自适应性

范式适应性速度原因
Agent Teams⭐⭐⭐ 高⭐⭐ 中Lead 动态决策，每步耗 token
Dynamic Workflows⭐⭐ 中⭐⭐⭐⭐⭐ 快脚本固定，编排不耗 token
OpenClaw Subagents⭐⭐⭐⭐ 最高⭐ 慢最灵活，但每步最贵
ACP协议层不做编排决策

最佳场景

范式适合不适合
Dynamic Workflows大规模搜索、审计、迁移、交叉验证需动态调整方向的探索
Agent Teams需 peer 对话的协同开发超大规模或一次性并行
OpenClaw Subagents结构化日常任务、实验迭代peer 间通信或超大规模
ACP异构框架互操作单框架内部协作（太重）

演进趋势

⬇ 单 agent 线性执行
⬇ 父子 subagent（OpenClaw）— 简单可靠，适合日常
⬇ Peer-to-peer 团队（Agent Teams）— 适中小团队协作
⬇ 脚本编排大规模并行（Dynamic Workflows）— 企业级规模化
⬇ 跨框架标准协议（ACP）— 异构生态互连
核心洞察： Dynamic Workflows 最重要的创新不是"1000 个 agent"，而是 "orchestration as code" 模式——将协调逻辑从模型上下文移到可版本控制的脚本。
  • ✅ 可审计——脚本是明文 JS，非黑盒
  • ✅ 可 rerun——同样脚本重跑结果一致
  • ✅ 可集成 CI/CD——脚本可纳入部署流水线
  • ✅ 成本可预测——编排开销固定
