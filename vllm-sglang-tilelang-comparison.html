vLLM vs SGLang vs TileLang — LLM 推理框架对比

三大推理引擎架构、性能、场景深度对比
Nemesis 2026-05-21 LLM 推理 模型服务

一、概览

三个项目处于不同技术栈层级：

项目定位代表作
vLLM完整的 LLM 服务/推理引擎PagedAttention, Continuous Batching
SGLang完整的 LLM 服务/推理引擎RadixAttention, 结构化输出
TileLangGPU 内核 DSLDeepSeek TileKernels

vLLM 和 SGLang 是竞争关系——都提供完整的端到端服务能力。TileLang 是底层算子开发工具，可与二者组合使用。

二、核心架构

vLLM

技术说明
PagedAttention核心创新。KV Cache 分页管理（类似虚拟内存），消除内存碎片
Continuous Batching请求到达即插入运行中批次，不等当前批次完全结束
Prefix Caching缓存公共前缀的 KV Cache
Chunked Prefill长 prompt 分块处理，避免独占 GPU 过久
CUDA Graph支持 CUDA Graph 加速 decode 阶段
V1/V2 架构MRV2 在 GB200 上带来 56% 吞吐提升

SGLang

技术说明
RadixAttention核心创新。基数树 KV Cache 复用，自动检测共享前缀。比 vLLM 的 prefix caching 更精细
Zero-overhead CPU SchedulerCPU 端调度开销近乎为零
Prefill-Decode Disaggregation预填充和解码分离，可独立扩展
Speculative Decoding原生支持推测解码
结构化输出原生支持 JSON Schema 约束解码
前端语言prompt 中嵌入编程逻辑（控制流、多模态、并行调用）

TileLang

  • 基于 Apache TVM 编译器基础设施
  • Pythonic DSL：用 Python 语法表达 GPU 内核，代码量减少 90%
  • Auto Tiling：自动选择最优 tile 配置
  • Auto TMA/WGMMA：自动利用 H100 硬件指令
  • Auto Pipelining：自动生成多级流水线调度
  • 不提供模型加载、batching、caching、HTTP API——纯内核编写工具

关键发现

  • SGLang 在中高并发下略优于 vLLM（~3%）
  • SGLang 在前缀复用场景（系统提示词、RAG 模板）优势可达 2-5x
  • 显存占用差异小于 4GB——引擎选择不如 max-model-len 和 gpu-memory-utilization 设置重要

六、独特特性

vLLM

  • 最广泛的模型架构支持（200+），社区首选测试平台
  • PyTorch Foundation 成员，生态整合最深
  • 多硬件后端最广（NVIDIA + AMD + TPU + Intel + AWS + Apple）
  • 70K+ Stars，2000+ 贡献者

SGLang

  • RadixAttention：前缀复用场景吞吐可超 vLLM 数倍
  • Prefill-Decode Disaggregation：prefill 和 decode 分配到不同 GPU
  • 结构化输出：原生 JSON Schema 约束解码，Agent 场景极有价值
  • RL 训练集成：被 AReaL、verl、Miles 等 RL 框架作为 rollout 后端
  • 宣称驱动全球 400,000+ GPU

TileLang

  • 80 行 Python 代码实现 FlashAttention
  • TVM 编译器后端：自动调度、内存布局、流水线优化
  • DeepSeek TileKernels：生产级内核库
  • 跨平台：同一代码编译到 CUDA、HIP、Metal
  • ICLR 2026 Oral 论文

八、总结

  • vLLM 是更安全的全能默认选择——模型广、文档好、社区大
  • SGLang 在特定场景（前缀复用、结构化输出、RL）有差异化优势
  • TileLang 定位完全不同——编写高性能 GPU 内核的 Python DSL
  • vLLM 和 SGLang 都在快速迭代，性能差距在缩小
