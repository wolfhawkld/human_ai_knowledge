LLM 生成矢量图的能力边界分析

整理日期：2026-03-17
参与者：Damon, Nemo
来源：对话讨论总结

背景

以 Simon Willison 维护的 "Pelican riding a bicycle" SVG benchmark 为切入点，探讨了当前 LLM 生成矢量图的能力边界，以及其对论文配图等实际应用场景的适用性。

1. 当前 LLM 的 SVG 生成能力

1.1 Benchmark 测试结果（2026年初）

Simon Willison 维护了 100+ 篇测试文章，追踪各模型的 SVG 生成能力。

最佳表现排名：

🥈 Gemini 3.1 Pro 2026-02-19 思考 323.9 秒，腿正确显示在两侧

1.2 能力评估

已解决的问题：

  • 基本的"能不能画"问题
  • 简单的空间组合
  • 常见物体的几何表达

仍然困难的问题：

  • 精确的数据驱动图形
  • 复杂的标注系统
  • 符合期刊标准的格式

2. 为什么论文配图比 Benchmark 更难

2.1 需求对比

维度 Pelican 骑车 Benchmark 论文 SVG 配图
精度要求 大概像就行 坐标/比例必须精确
数据驱动 无 需要与数据/公式对应
风格约束 无 期刊模板、字号、配色规范
复杂度 2 个主体 多组件、多层次、标注系统
可迭代性 不重要 需要反复微调
专业性 无 领域知识、设计原则

2.2 实际问题

问题 1：模型不知道你的数据

"画一个系统架构图"
→ 画出通用架构，不是 YOUR 系统

问题 2：坐标和尺寸是噩梦

<!-- 模型生成的 -->
<rect x="120" y="85" width="200" height="100"/>
<!-- 实际需要的 -->
<rect x="118.5" y="83.2" width="203.7" height="98.4"/>

问题 3：无法处理复杂标注

  • 图例位置要避开主要内容
  • 标签要跟箭头/连线对齐
  • 多个子图要布局协调

问题 4：缺乏设计能力

  • 配色方案
  • 视觉层次
  • 留白和平衡

3. 为什么 Scaling 不能直接解决

3.1 核心问题：不是"聪明程度"，是"训练分布"

LLM 训练数据分布：
- 大量文本 ✓
- 大量代码 ✓
- 大量图像-文本对 ✓
- 高质量 SVG 数据 ✗ （稀少）
- SVG + 视觉对应关系 ✗ （几乎没有）

类比：即使 GPT-10 再聪明，如果没见过医疗数据，也做不了医生诊断。

3.2 能力类型与 Scaling 的关系

能力类型 是否可通过 Scaling 突破
语言理解 ✅ 可以
推理能力 ✅ 可以
代码生成 ✅ 可以（如果训练数据多）
SVG 生成 ❌ 需要专门数据
精确控制 ❌ 架构问题，不是规模问题

4. SVG 问题比 CLIP 更复杂

4.1 CLIP 解决的问题

文本 ──CLIP── 图像
        ↓
   共享嵌入空间（双模态对齐）

CLIP 解决了文本和图像的理解和检索，但不是生成。

4.2 SVG 的三模态问题

文本 ──?── SVG代码 ──渲染── 图像
              ↓
         坐标/路径/样式

三层鸿沟：

  1. 文本 → SVG 代码（语言模型擅长）
  2. SVG 代码 → 视觉想象（模型看不见）
  3. 视觉想象 → 精确代码（需要逆向工程）

4.3 核心难点对比

CLIP 的世界 SVG 的世界
像素是连续的 SVG 是离散代码
大量现成数据 高质量 SVG 稀缺
理解任务 生成 + 精确控制
容错性高 坐标差 1px 就错

4.4 可能的突破口

  1. SVG 预训练语料库 - 像 LAION 之于图像
  2. 渲染感知训练 - 让模型在训练时"看到"自己生成的 SVG
  3. 分层生成 - 先布局后细节，类似人类设计流程
  4. 工具增强 - 模型调用专业 SVG 工具，而非手写代码

5. Agent 框架能带来什么

5.1 Agent 循环的价值

传统 LLM：
描述 → SVG → 结束

Agent 循环：
描述 → SVG → 渲染 → 看图 → 评估 → 修改 → 渲染 → ...
                    ↑                          |
                    |__________________________|

核心增益：视觉反馈闭环

5.2 实际能做的事

✅ 可以做到：

  1. 生成 SVG 骨架
  2. 调用渲染工具（浏览器/puppeteer）
  3. 截图 → 发给视觉模型
  4. 视觉模型评估问题
  5. 根据反馈修改
  6. 重复直到满意

⚠️ 部分能做到：

  • 简单架构图：反复迭代后可能合格
  • 流程图：节点和连线相对规则
  • 示意图：不需要精确数据

❌ 仍然困难：

  • 数据驱动的图表：需要精确对应数据
  • 复杂标注系统：字体、位置、层级
  • 期刊标准格式：尺寸、分辨率、配色规范

5.3 Agent 的关键限制

限制 1：视觉模型不擅长精确测量

视觉模型："这个图看起来不对齐"
但说不出："x 坐标应该从 120 改成 135.5"

限制 2：迭代成本高

每轮迭代：生成 → 渲染 → 评估 → 修改
论文配图可能需要 10-50 轮迭代

限制 3：缺乏设计知识

模型知道：SVG 语法
模型不知道：视觉层次原则、专业配色、期刊排版规范

6. 成本效益分析

6.1 成本对比

人工绘图：
思考 → 设计 → 画图 → 微调
时间：1-4 小时（熟练者）
结果：精确可控

Agent 迭代：
描述 → 生成 → 渲染 → 评估 → 修改 → ...
时间：可能更久
结果：还不一定满意

结论：当前阶段，复杂矢量图 AI 反而更贵。

6.2 什么时候值得用 AI

场景 是否值得用 AI
论文核心配图 ❌ 自己画
快速出草稿讨论 ✅ AI 生成
批量生成类似图形 ✅ AI 批量处理
格式转换（PNG→SVG） ✅ AI 辅助
简单图标/符号 ✅ AI 生成
需要精确数据的图表 ❌ 用 Matplotlib/TikZ

7. 结论与建议

7.1 核心判断

  1. 没有专门 SVG 数据训练 → 不太可能生成高质量矢量图
  2. 模型能力增长 → 不会自动解决这个领域问题
  3. Agent 框架 → 能改善但不是银弹
  4. 迭代成本 → 当前阶段往往比人工更高

7.2 AI 的真正价值

不是：替代专业绘图
而是：
├── 快速原型（讨论用）
├── 批量处理（重复劳动）
├── 格式转换
└── 辅助灵感

7.3 实践建议

对于论文配图：

  1. 核心图表：使用 TikZ、Matplotlib、draw.io 等专业工具
  2. 快速原型：可用 AI 生成讨论
  3. 批量处理：AI 辅助格式转换和批量生成
  4. 最终出品：人工把控设计和细节

工具选择优先级：

精确需求 → 代码生成工具（TikZ/Matplotlib）
设计需求 → 专业设计工具（Figma/Illustrator）
快速原型 → AI 生成
批量处理 → AI + 自动化脚本

8. 未来展望

如果 SVG 生成能力要突破，需要：

  1. 专门数据集 - 高质量 SVG + 文本描述 + 渲染图像的对应数据
  2. 三模态对齐 - 文本-SVG代码-视觉的三方嵌入空间
  3. 架构创新 - 可能需要"视觉反馈 + 代码生成"的新范式
  4. 工具集成 - LLM 调用专业矢量工具，而非手写代码

这些需要专门的研究和工程投入，不太可能通过单纯 Scaling 自然解决。
